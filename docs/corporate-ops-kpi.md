# コーポレートサイト：更新ルールと KPI

社内向けの運用メモです。リニューアル後の継続改善（掲載内容の鮮度・計測）に使います。

## 更新ルール

| 領域 | 推奨頻度 | 担当イメージ | 備考 |
|------|-----------|--------------|------|
| ニュース・ブログ | 月1回以上（可能なら隔週） | マーケ／事業 | トップの「ニュース・ブログ」は Markdown の最新日付で自動反映。公開後は `npm run build` で本番反映。 |
| 企業情報（沿革・資本・取引先など） | 四半期、または変更発生時 | 経営／バックオフィス | `CompanyProfileSection` と公式デッキ・登記情報を突合。 |
| サービス説明・事例 | 四半期、またはプロダクトリリース時 | 事業 | `ServiceSection`・各サービスページと `COPYWRITING.md` を揃える。 |
| メタデータ・OG | ブランドコピー変更時 | Web | `lib/site-metadata.ts` を単一ソースとして更新し、`StructuredData` と矛盾がないか確認。 |
| お問い合わせ種別・文言 | チャネル追加時 | 営業／採用 | `ContactSection` と GAS シートの列定義を同期（`inquiryType` の値）。 |

## KPI（目安）

PV 偏重ではなく、**コンバージョンと主要導線**を優先します。

1. **お問い合わせ件数**（フォーム送信＋ `hello@amber-inc.com` の手動カウントを週次で合算してもよい）
2. **ブログ／ニュース経由の遷移**（サイト内: トップ「すべての記事を見る」クリック、記事からの CTA）
3. **企業情報ページ（`/company`）のエンゲージメント**（滞在時間・スクロール深度。Vercel Analytics 等で可能な範囲で）
4. **サービス下層への遷移**（`/service/consulting` 等へのクリック）

四半期ごとに、上記をスプレッドシートまたは Notion に1行まとめし、前年同期・前期比でレビューする運用を推奨します。

## 品質チェック（任意・リリース前）

- ローカルで `npm run build` が通ること
- 主要ページ（`/`, `/company`）で見出し階層が論理的であること（ページごとに1つの `h1`）
- Lighthouse（Performance / Accessibility）で明らかな退行がないこと。動きを減らすユーザー設定ではヒーローの WebGL 背景を読み込まない実装になっています。
